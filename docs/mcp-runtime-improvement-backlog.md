# Godot MCP 运行时改进清单

## 文档目的

这份文档记录本项目在使用 Godot MCP 做运行时观察、结构化控制与自动验收时，**实际遇到的问题**、**优先级排序后的改进项**，以及每项能力的**落地验收标准**。

它服务于以下场景：

- 后续继续增强 `addons/godot_mcp/`；
- 为 MCP 仓库提 issue / roadmap；
- 帮助代理在自动验收失败时快速判断是项目问题还是 MCP 能力边界；
- 为将来的 GUT / MCP / 人工验收协同方案提供基础约束。

---

## 当前结论

截至本次暂停系统验收实践，Godot MCP 的能力可以概括为：

- **观察能力较强**：能看日志、报错、截图、场景树、节点属性、节点方法；
- **结构化控制可用**：能启动/停止项目、调用简单方法、设置简单属性、实例化和删除节点；
- **自动验收能力不够稳定**：复杂参数传递、复杂运行时编排、精确 UI 驱动、帧级验证仍然容易失败。

一句话总结：

> MCP 现在已经很适合“看”和“改”，但距离“稳定做自动验收与复杂运行时编排”还有明显缺口。

---

## 本次实践中实际遇到的问题

以下问题并不是推测，而是本项目在实现“UE5 语义真暂停”并尝试用 MCP 做运行时验收时真实踩到的。

### 1. 复杂参数传递不可靠

最明显的问题是 `Vector2`。

当通过 MCP 调用诸如：

- `spawn_projectile(position, direction, ...)`
- `spawn_enemy_projectile(position, direction, ...)`
- `setup(direction, speed, damage, source)`

这类方法时，MCP 传过去的结构在 Godot 侧会被解释成 `Dictionary`，而不是 `Vector2`，最终报错：

- `Cannot convert argument from Dictionary to Vector2`

这会直接阻断大量高价值运行时操作。

### 2. 命令存在随机超时

本次执行中，以下命令都出现过“有时成功、有时超时”的情况：

- `remove_node`
- `get_nodes_in_group`
- `set_property`
- `screenshot`
- `eval`

这意味着超时并不总是调用者写错了，而是 MCP 链路本身在复杂运行态下稳定性不足。

### 3. `eval` / 临时脚本执行链不够稳

本次为了加速验证，尝试通过 MCP 执行少量运行时代码。期间实际出现过：

- `Mixed use of tabs and spaces for indentation`

该问题来自 MCP 注入的临时代码执行链，而不是项目业务脚本本身。

结论是：`eval` 目前不能作为一个可靠的兜底能力。

### 4. 纯屏幕坐标驱动 UI 交互不够可靠

像下面这种流程：

- 点击地图节点；
- 点击确认；
- 进入战斗；
- 开始波次；

理论上可以完全靠截图 + 屏幕坐标完成，但实际很容易失败。原因可能包括：

- 控件命中区域不精确；
- UI 刷新与点击时机不同步；
- 坐标命中可见元素，但没有命中真正的交互节点；
- 某些界面遮罩或状态切换会吞掉输入。

因此 MCP 当前更适合“结构化控制”，不适合承担高保真玩家模拟。

### 5. 运行态状态读取有时不够一致

在复杂流程里，偶尔会出现：

- 一个接口返回当前状态是 A；
- 另一个接口看起来还停留在 B；
- 返回格式不统一，导致自动验收脚本很难做稳定判断。

这类问题在做“暂停前运动 / 暂停中冻结 / 恢复后继续”这种时序敏感验证时尤其明显。

## 优先级总表

| 优先级 | 改进项 | 影响面 | 为什么优先 |
|---|---|---|---|
| P0 | 强类型参数传递 | 极高 | 直接影响方法调用是否可用 |
| P0 | 超时与失败诊断 | 极高 | 当前失败后信息不足，难快速定位 |
| P0 | 稳定的 `eval` / 脚本执行 | 高 | 是复杂验收和排障的重要兜底 |
| P1 | 控件级 UI 操作 | 高 | 可显著提升菜单/地图/设置页自动验收稳定性 |
| P1 | 更强的节点查询与过滤 | 高 | 提高运行时定位、断言与问题收敛效率 |
| P1 | 帧级状态采样 / diff | 高 | 直接服务暂停、动画、移动、物理冻结等验证 |
| P1 | 增量日志与过滤能力 | 中高 | 提高长流程调试可读性 |
| P2 | 录制 / 回放式验收 | 中 | 有价值，但不如基础稳定性紧急 |
| P2 | 节点状态快照与差异比较 | 中 | 对复杂问题定位很有帮助 |
| P2 | 进程模式专项诊断 | 中 | 对 Godot 暂停语义 / process_mode 问题非常有针对性 |

## P0：必须优先补齐的能力

### 一、强类型参数传递

#### 问题

当前 `call_method` 与部分属性设置对 Godot 复杂类型支持不可靠，尤其是：

- `Vector2`
- `Vector3`
- `Color`
- `Rect2`
- `Transform2D`
- `NodePath`
- Typed `Array`
- Typed `Dictionary`

#### 改进目标

MCP 协议层必须能**显式表达 Godot 值类型**，而不是让调用端和服务端各自猜测。

推荐方向：

```json
{
  "_godot_type": "Vector2",
  "x": 100,
  "y": 200
}
```

或更统一一点：

```json
{
  "type": "Vector2",
  "value": { "x": 100, "y": 200 }
}
```

#### 验收标准

至少以下调用必须稳定通过：

- `spawn_projectile(Vector2, Vector2, float, float, Object)`
- `setup(Vector2, float, float, Object)`
- `set_property("global_position", Vector2)`
- `set_property("modulate", Color)`

并且失败时必须清晰提示“类型不匹配的具体参数位”。

---

### 二、超时与失败诊断能力

#### 问题

当前报 timeout 时，缺少足够上下文，无法快速判断到底是：

- 命令排队超时；
- Godot 主线程阻塞；
- 节点路径解析卡住；
- 目标方法执行异常；
- 命令返回值序列化失败。

#### 改进目标

每次失败都至少返回：

- 命令类型；
- 目标节点；
- 目标方法 / 属性；
- 失败阶段；
- Godot 侧错误消息；
- 是否是 timeout；
- timeout 发生前最后执行到的步骤。

#### 验收标准

出现超时后，日志或返回结果中应能明确区分：

1. **命令未送达**；
2. **命令已送达但执行阻塞**；
3. **命令执行报错**；
4. **命令执行成功但返回值封送失败**。

---

### 三、稳定的 `eval` / 临时脚本执行能力

#### 问题

`eval` 目前不够稳定，甚至会被缩进、格式、临时代码拼装问题影响，导致它不能承担“复杂排障兜底工具”的角色。

#### 改进目标

提供一个可靠版本的脚本执行接口，具备：

- 自动统一缩进；
- 明确返回值；
- 明确异常栈；
- 支持 typed args；
- 支持多行脚本；
- 支持只读执行模式。

#### 验收标准

以下场景必须可用：

- 读取任意节点属性并组合返回；
- 在运行态内构造 `Vector2` 并调用业务方法；
- 捕获业务脚本报错并返回完整栈；
- 连续执行多次不因缩进问题随机失败。

---

## P1：应尽快补齐的运行时验收能力

### 一、控件级 UI 操作

#### 问题

当前大量 UI 验证只能依赖屏幕坐标点击，这种方式太脆弱。

#### 改进目标

允许通过节点路径或控件实例直接：

- 点击按钮；
- 触发 `pressed`；
- 设置 / 获取选中状态；
- 读取 `disabled`、`visible`、`text` 等关键属性；
- 对 OptionButton / CheckButton / Tree / TabContainer 做结构化操作。

#### 理想接口示例

- `ui_click_node("/root/Main/PauseMenu/.../ResumeButton")`
- `ui_get_state("/root/Main/MapScreen/.../ConfirmButton")`
- `ui_select_tab("/root/Main/SettingsMenu/.../Tabs", 2)`

#### 验收标准

能稳定完成以下自动化流程而不依赖屏幕坐标：

- 打开暂停菜单并点击继续；
- 在主菜单点击开始新游戏；
- 在设置页切换某个选项；
- 读取地图确认按钮当前是否可按。

---

### 二、更强的节点查询与过滤

#### 问题

当前能查节点，但不够“工程化”，无法高效回答“哪些对象还在动”“哪些 projectile 处于 active 状态”这类问题。

#### 改进目标

增加以下查询能力：

- 按脚本类名查节点；
- 按脚本资源路径查节点；
- 按 group + class 组合过滤；
- 按属性值过滤；
- 按 `process_mode` / `is_processing` / `is_physics_processing` 过滤。

#### 验收标准

至少可以稳定回答：

- 当前场景里有哪些 `Projectile`；
- 哪些节点属于 `enemies` 且 `visible = true`；
- 哪些节点在 `get_tree().paused = true` 时仍在 processing；
- 哪些节点脚本路径来自 `res://scripts/projectile.gd`。

---

### 三、帧级状态采样与差异比较

#### 问题

像“暂停是否冻结”“动画是否停止”“移动是否继续推进”这类问题，用单次截图或单次属性读取很难做强判定。

#### 改进目标

增加帧级采样工具，例如：

- 连续 N 帧采样某节点位置；
- 连续 N 帧采样某属性；
- 直接比较暂停前 / 暂停中 / 恢复后的数值变化。

#### 理想接口示例

- `sample_property(node, property, frames)`
- `compare_property_during_pause(node, property, before, paused, resumed)`

#### 验收标准

能自动回答：

- 某投射物在暂停期间位置是否保持不变；
- 某敌人恢复后是否继续移动；
- 某动画播放节点在暂停前后播放状态是否变化。

---

### 四、增量日志与过滤能力

#### 问题

当前日志能看，但不够适合长流程调试和自动验收。

#### 改进目标

支持：

- 只读“自上次调用以来的新日志”；
- 只读 error / warning；
- 按节点名或脚本路径过滤；
- 按时间窗口过滤。

#### 验收标准

在一次完整战斗流程里，调用者能只抓取：

- 本波次新增日志；
- 仅 `ProjectileSpawner` 相关日志；
- 仅错误和 warning。

## P2：中期增强项

### 一、录制 / 回放式验收

适合把一套真实操作沉淀成可重复回放的流程，用于回归验证。价值很高，但前提是基础控制链已经足够稳定。

### 二、节点状态快照 / diff

适合做复杂问题排查，比如：

- 暂停前后谁的属性变了；
- 某次状态切换新增 / 删除了哪些节点；
- 波次开始前后 active enemy 数量是否符合预期。

### 三、进程模式专项诊断

对 Godot 项目尤其有价值。理想上 MCP 应该能直接回答：

- 哪些节点是 `PROCESS_MODE_ALWAYS`；
- 哪些节点是 `PROCESS_MODE_PAUSABLE`；
- 哪些节点会因父节点继承到错误的 process mode；
- 哪些节点在 paused 状态下仍然跑 `_process` / `_physics_process`。

这类能力对“真暂停”问题的定位效率会非常高。

---

## 推荐演进顺序

建议按以下顺序推进 MCP 改造：

### 第一阶段：先修“能不能用”

1. 强类型参数传递；
2. timeout / error 诊断；
3. 稳定 `eval`。

### 第二阶段：再修“能不能稳定验收”

4. 控件级 UI 操作；
5. 节点查询增强；
6. 帧级状态采样；
7. 增量日志过滤。

### 第三阶段：最后补“验收基础设施化”

8. 录制 / 回放；
9. 状态快照 / diff；
10. 进程模式专项诊断。

---

## 对本项目的直接收益

如果上述能力补齐，本项目以下场景会明显受益：

- 暂停系统自动验收；
- GUIDE 输入上下文切换验证；
- 地图 → 战斗 → 结算完整流自动检查；
- 波次状态机验证；
- 商店、设置、暂停菜单的 UI 回归测试；
- projectile / enemy / ship 的运行时状态定位。

尤其是本项目这类：

- 运行时场景驱动；
- 依赖 autoload 协调；
- 大量行为由 process / physics_process 驱动；
- 目前还没有完整传统自动化测试体系；

MCP 的稳定性提升，会直接转化成项目的交付效率和回归效率。

---

## 当前建议结论

如果只能先做最少的改造，建议先只做三件事：

1. **把复杂参数类型支持做对；**
2. **把 timeout / error 诊断做清楚；**
3. **把 `eval` 变成可靠工具。**

只要这三项到位，MCP 就能从“调试辅助工具”升级成“可承担复杂运行时验证的工程工具”。
